# NEXZ 髮夾補貨監控 — 實作說明

## 核心關鍵：避開網頁爬蟲的脆弱性

搜尋頁面（`/en/search?q=...`）是把「Sold Out（售完）」以 HTML 文字呈現。直接爬這段文字很
脆弱——用字會變、會有翻譯、還可能是延遲載入的內容。但這是一個 **Shopify** 商店，而 Shopify
為每一項商品都提供一個隱藏的 JSON 端點：

```
https://jypj-store.com/products/<handle>.js  ->  {... "available": true/false ...}
```

這個 `available` 布林值，正是商店自己「加入購物車」按鈕所使用的同一個旗標。在採用之前，已先用
一個確定有貨的商品（FOX2Y -> true）和一個售完的商品（-> false）驗證過。因此監控程式不去解析
網頁，而是輪詢 7 個極小的 JSON 端點——快速、明確、且與語言無關。

### 髮夾 -> handle 對照表

| 髮夾     | Handle        |
|----------|---------------|
| YUTiE    | nx26-lt0-0037 |
| PPOMOYA  | nx26-lt0-0031 |
| HYUROMI  | nx26-lt0-0036 |

（FOX2Y / `nx26-lt0-0038` 依照需求刻意排除。）

## 為什麼不用 WebFetch 工具？

你最初給的網址是經過 `WebFetch` 工具，而它會把每個網址的結果**快取 15 分鐘**。用 10 秒一次
去輪詢一個快取 15 分鐘的結果，只會一直拿到舊資料。因此監控程式改用原生 `curl`，每一輪都直接
打到即時的伺服器。

## 架構 — 三個部分

1. **`restock_monitor.sh`** — 監控迴圈：
   ```
   每 10 秒：
       對 7 個 handle 各做一次：
           curl 該 .js 端點，grep "available":true/false
       若有任一為 true：
           寫入紀錄、寄出 Email、exit 0
       否則：
           寫入「仍售完」、sleep 10
   安全上限：4320 次（約 12 小時）
   ```

2. **`send_mail.py`** — Gmail SMTP 寄信程式。以 SSL 連到 `smtp.gmail.com:465`，用應用程式
   密碼登入，寄出純文字的 `EmailMessage`。主旨與內文用參數傳入，因此同一支程式同時供測試信與
   正式通知信使用。

3. **`mail.cfg`** — 帳密設定檔（`chmod 600`），由 `send_mail.py` 讀取。與程式碼分離，避免把
   機密寫死在程式裡。

## 兩層通知（縱深防禦）

- **主要 — 自動寄信：** 一偵測到有貨，bash 腳本「自己」就呼叫 `send_mail.py`。即使 Claude
  在閒置、或工作階段在休眠，這封信仍會寄出——不依賴助手是否在線。
- **備援 — 聊天室內通知：** 腳本以背景任務執行。背景程式結束時，harness 會重新喚醒 Claude；
  而此腳本只有在補貨（或達到 12 小時上限）時才會結束。因此補貨同時也會喚醒 Claude 在聊天室
  通知你。

一次補貨會同時觸發兩條路徑；寄信失敗不會影響聊天室通知，反之亦然。

## 幾個可靠性設計決策

- 每次 curl 都加 `--max-time 8`——卡住的請求不會拖垮整個迴圈。
- 故障時的安全方向：若商店做了速率限制或回應錯誤，grep 找不到 `"available":true`，會被視為
  「仍售完」。最糟只是「漏掉一次檢查」，絕不會誤寄「假補貨」通知。
- 僅附加（append-only）的紀錄檔（`restock.log`）——讓「目前狀態」可由最近幾行加上計數器回答。
- 重啟紀律：加入寄信功能時，先停掉舊程式再重新啟動，因此不會有兩個監控同時重複寄信。

## 誠實說明限制

- 它只在工作階段／機器持續運作期間存活——無法在重開機後自動復原。用 cron 排程或 systemd
  服務可解決。
- 應用程式密碼以明文存放在 scratchpad 的 `mail.cfg`。用完請到
  https://myaccount.google.com/apppasswords 撤銷。
- 10 秒輪詢是固定間隔，並非事件驅動——真正補貨與通知之間最多有 10 秒延遲。Shopify 對買家
  並未提供公開的推播／webhook。
